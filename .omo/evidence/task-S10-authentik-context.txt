## S10 必须行为

1. Authentik OIDC subject 必须映射为 `UserMirror.authentik_user_id`，不能使用邮箱、用户名、工号或 DingTalk ID 作为规范授权标识。
2. webhook 和定时同步两条路径都必须调用同一套 payload 解析和 upsert 语义。
3. Authentik payload 的用户状态映射到 `UserMirror.status`：active 保留授权；inactive、disabled、departed 进入 EasyAuth 非 active 状态。
4. 用户变为非 active 时，必须触发该用户所有 current active grants 的撤销，每条受影响 grant `version += 1`，`status=revoked`，`is_current=False`。
5. 同步撤权必须写审计：至少 `user_departure_detected` 和每条 grant 的 `grant_revoked`。`grant_revoked` 可复用现有 `GrantService.revoke_grant` 的审计格式，actor 建议为 `actor_type="authentik"` 或架构允许的系统身份。
6. 重复 webhook 或重复定时同步必须幂等：已 disabled/departed 且已撤销的 grant 不应再次递增 version，也不应重复写 grant revoke 审计。
7. 没有真实 Authentik 凭据时，测试必须使用夹具 payload 和模拟 client；不得要求真实网络环境。

## 建议 worker 先写的 RED/PIN 测试

1. `tests/unit/integrations/authentik/test_payloads.py`
   - Authentik user write event payload 中 `user.pk` 或 OIDC subject 字段解析为 `authentik_user_id`。
   - `user.email`、`user.username`、`context.name/email/username` 映射到 `UserMirror` 可变展示字段，但不替代 subject。
   - `context.attributes` 中的部门、工号、DingTalk 字段存在时能映射；缺失时落为空字符串或保持既有值，按实现契约固定。
   - `is_active=True` 映射 `active`；`is_active=False` 映射 `disabled` 或 `departed` 中的明确一种；payload 明示 departed 时映射 `departed`。
   - malformed payload：缺 subject/user pk、user 不是对象、email 类型错误、attributes 不是对象、is_active 缺失或非布尔、payload 同时给出冲突身份字段，必须返回类型化错误或拒绝，不要静默创建错误用户。

2. `tests/unit/accounts/test_services.py`
   - `AuthentikSyncService.upsert_user(payload)` 首次创建 `UserMirror`，重复同 subject 更新 name/email/department/status 但不新建用户。
   - subject 变化时不得把既有用户改成新 `authentik_user_id`；应创建新用户或拒绝，具体按 parser 契约测试固定。
   - active -> disabled/departed 触发撤权；disabled/departed -> active 只更新用户状态，不自动恢复 grants。
   - 重复 disabled/departed 同步幂等，不重复递增 grant version。

3. `tests/unit/grants/test_services.py`
   - 新增 `GrantService.revoke_for_user(user, reason, actor)`：同一用户多个 app 的 current active grants 全部撤销。
   - 非 current、已 revoked、已 expired grant 不受影响。
   - 每条撤销 grant version 递增一次，写一条 `grant_revoked`。
   - 没有 active grants 返回空列表，不写 grant revoke 审计。

4. `tests/integration/authentik/test_sync.py`
   - webhook path 和 scheduled sync path 都调用同一 upsert 行为。
   - Authentik inactive/disabled/departed payload 后，权限查询对该用户返回空 roles/permissions，version 为撤销后的最新 version。
   - 多 app 授权场景下每个 app 查询都能看到撤权后的空权限。

5. PIN 测试建议

## 不能越界实现的内容

- 不要实现 S12 Authentik OIDC 登录、portal session、login callback 或 session middleware。
- 不要实现真实 Authentik 网络 client、真实凭据读取、真实 issuer/client secret 配置校验。
- 不要实现 S15 定时过期清理；S10 只需要定时同步入口或任务壳调用模拟 client，不处理 grant expiration 扫描。
- 不要绕过 `GrantService` 直接批量 update grants，否则会丢失 version、审计和幂等语义。
- 不要把 email、username、employee_number、DingTalk ID 当作权限查询的公共 user_id。
- 不要把离职用户物理删除；`UserMirror` 设计为镜像记录且禁止物理删除。

## 风险和未知点

- Authentik payload 字段形态需要在实现中类型化固定。官方依据已确认 user write event payload 含 `user.pk/email/username`、`context.attributes`、`context.name/email/username`，但 OIDC subject 在 webhook payload 中的具体字段可能需要通过夹具约定；项目要求最终映射为 `authentik_user_id`。
- `inactive` 到 EasyAuth 的 `disabled` 还是 `departed` 需要实现者明确规则。架构只说 inactive/disabled 会映射为不可保留授权状态，S10 标题强调离职撤权；建议用 payload 明示 departed 时映射 `departed`，否则 `is_active=False` 默认 `disabled`。
- 架构契约要求 `GrantService.revoke_for_user`，现有代码只有 `revoke_grant(user, app, ...)`。worker 需要补一个用户级撤权边界，且保持事务和 select_for_update 行为。
- `user_departure_detected` 审计目前没有现成封装；可以直接复用 `AuditService.record`，但 target_type/target_id/metadata 需要由测试固定。
- 目前 `src/easyauth/accounts/services.py`、`src/easyauth/integrations/authentik/`、`src/easyauth/tasks/authentik.py` 不存在；新增文件应遵守当前 Python typing 风格和中文文档规则。
- 定时同步入口如何接入任务系统尚未有 Celery 基线代码。S10 可以先提供可测试函数/任务壳和模拟 client，不应引入真实 worker 配置。

## 建议实现顺序

1. 先新增 Authentik payload parser 的 RED 测试，固定 subject、展示字段、状态映射和 malformed payload。
2. 再新增 `GrantService.revoke_for_user` 的 RED 测试，复用现有单 app revoke 语义。
3. 实现 `GrantService.revoke_for_user`，让它按用户遍历 current active grants 并调用或等价复用单 grant 撤销逻辑。
4. 新增 `AuthentikSyncService.upsert_user`，在一个事务中 upsert `UserMirror`，状态变为非 active 时调用 `GrantService.revoke_for_user` 并记录 `user_departure_detected`。
5. 新增 webhook handler 和 scheduled sync 入口的薄封装测试，确保两条路径共享同一 upsert 服务。
