# 实施与发布计划

> 第 7 篇。模块归属、迁移步骤、灰度/回滚、测试要点（含契约测试）。

## 1. 模块归属决策

| 能力 | 归属 | 理由 |
|---|---|---|
| 通知（模型/服务/迁移） | **新建 Django app `easyauth.notify`** | 拥有自己的持久状态（两张表）、状态机与生命周期任务；与 webhooks 是平行的出站管道（第 3 篇 §0 论证过不能塞进 webhooks）；独立 app 让迁移、测试目录（`tests/unit/notify/`）与未来扩展（多渠道）边界干净 |
| 通知 Celery 任务 | `src/easyauth/tasks/notify.py` | 仓库惯例：任务薄壳统一放 `easyauth/tasks/`（webhooks/lifecycle/connectors 均如此），业务编排留在 `notify/services.py` |
| 通知/目录 API 视图 | `src/easyauth/api/notify_views.py`、`src/easyauth/api/directory_views.py` | 公共 API 的函数视图全部集中在 `easyauth/api/`（现有 views/approval_views/manifest_sync_views），新端点跟随；沿用「函数视图 + 手动认证/限流 + build_error_response」风格，不引入 DRF ViewSet |
| 目录读逻辑 | `accounts` 镜像 + `api/directory_payloads.py` + `accounts/directory_snapshot.py` | 目录仍是镜像读投影，但镜像已增加联系字段、tombstone 与 generation；快照服务生成多企业权威性元数据 |
| 能力开关 | `applications` | `AppCapability` 由超管开通，单 credential capabilities 由 App owner 授权，两层逻辑与 |
| 通知通道 | `applications.AppNotificationChannel` | 每 App 独立、版本化 app_key/app_secret/agent_id，消息受理时冻结 channel |
| 钉钉发送方法 | `integrations/dingtalk/api_client.py` | 每次使用冻结通道凭据构造 client；oapi 例外边界仅限工作通知端点 |

## 3. 灰度与回滚

**灰度天然按 app 进行**——能力默认关闭：

| 阶段 | 动作 | 验证 |
|---|---|---|
| G0 上线 | 全部代码/迁移入生产，无 app 开通能力 | 现有端点回归无感知；beat 新任务空转（无 sent 行时 reconcile 是零查询） |
| G1 冒烟 | 建一个内部测试 app，开通 App 能力、授权测试 credential、配置 App 通道 | ① 通道连通性；② 三种模板手机/PC 展示与深链；③ 回执只对明确名单升级 delivered |
| G2 首个消费方 | 为 `easyproject` 开通，观察一周投递大盘 | 失败率、throttled 频次、配额水位 |
| G3 推广 | EasyTrade 等迁移目录消费，逐个开通 | — |

**回滚路径**（按事故等级从轻到重）：

1. 单 app 出问题 → console 关其能力（立即 403，进行中消息继续收敛，无残留）；
2. 通知管道出问题 → 关 beat 两任务 + 停 `notify` 队列 worker；已受理消息停在
   pending/sending，恢复后由 lease 过期机制自动续投（outbox/claim 设计的固有能力）；
3. 代码级回滚 → 回滚部署即可；`notify` 表与索引**不回滚迁移**（新表不影响旧代码），
   杜绝丢数据操作。

**发布前检查单**：

- [ ] 确认企业钉钉版本（标准版有 5000 次/月 API 总额度且工作通知计入，第 4 篇
      §4.2；标准版则压低各 app 配额并评估关闭对账）；
- [ ] 目标 App workspace 已由 owner 配置并测试 active 版本化钉钉通知通道；
- [ ] 超管已开通 App capability，且测试 credential 只获授必要 capability；
- [ ] 依赖健康检查已能报告「通知凭据缺失」（第 3 篇 §8）；
- [ ] `notify` 队列 worker 与现有 `webhooks` 队列同机部署编排已更新（deploy/）。

## 4. 测试要点

### 4.1 单元测试（`tests/unit/notify/`、`tests/unit/api/`、`tests/unit/applications/`）

- **收件人解析**：裸 user_id / `dt:` 引用 / 两者指向同一人合并 / 未知引用 /
  无钉钉绑定 / departed 用户 → 各自的终态与 error_code；
- **受理幂等**：同 dedup_key 同载荷 → 200 复用；同 key 异载荷 → 409；并发双写
  靠 DB 唯一约束兜底（捕获 IntegrityError 转幂等命中）；
- **msg 组装**：三模板的钉钉 JSON 结构、2048 字节边界（含多字节字符）、
  action_card 必填校验、deeplink 两种协议前缀校验；
- **状态机**：批次部分成功/全失败/QPM 频控/网络中断各路径的收件人与消息聚合状态；
  重试排程写出的 outbox event_key 与 countdown；耗尽 → EXHAUSTED + 审计；
- **对账**：getsendresult 四类名单 → 状态映射（含 143105/143106 区分）、
  无明确名单和 >24h 都保持 `sent`、reconcile 关闭开关；
- **claim/lease**：双执行体抢占互斥、lease 过期接管（照抄 webhooks 既有测试形态）；
- **配额与限流**：日配额边界（自然日翻转）、429 带 Retry-After；
- **能力开关**：App 层或 credential 层任一缺失都 403，两层动作审计；
- **目录过滤**：q/department_id/manager_id/include_inactive 组合、联系字段/tombstone、
  多企业 snapshot 新鲜度、snapshot_id 固定分页与 409 重拉。

### 4.2 契约测试（保证「文档即实现」）

- 为第 1 篇的**每个 JSON 示例**写断言测试：按示例构造数据，逐字段比对响应结构
  （风格对齐 `tests/unit/api/` 现有做法）；错误路径覆盖 §0.3 全表；
- SDK 侧：`sdk/python/tests/` 用 stub HTTP server 回放同一组契约 JSON，
  验证 7 个新方法的 URL/query/body 组装与透传——服务端与 SDK 共享
  同一份契约样例（考虑放 `tests/contract_samples/*.json` 供两侧引用）。

### 4.3 集成/手工

- 本机 Authentik 联调环境跑一轮真实目录同步后核对目录端点输出与 console 人员页一致；
- G1 冒烟中的真实钉钉发送三模板 + 撤回场景不测（非目标）；
- Celery 全链路：受理 → outbox 扫描 → worker 投递 → 对账升级，用
  `EASYAUTH_OUTBOX_DISPATCH_SECONDS=1` 压缩等待。
