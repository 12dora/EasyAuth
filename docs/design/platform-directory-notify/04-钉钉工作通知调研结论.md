# 钉钉工作通知调研结论（API 版本 / 凭据 / 频控 / 卡片类型）

> 第 4 篇。调研日期 2026-07-16，全部限制数字附官方文档出处；本篇是第 1/2/3 篇中
> 钉钉侧数字的唯一事实源。

## 1. API 版本结论：只有旧版 asyncsend_v2，无新版替代

- 官方《工作通知概述》的 API 总览中，工作通知全部 6 个接口（发送/更新状态栏/
  查进度/查结果/撤回/模板发送）的「API 版本」列均标注**旧版**（oapi.dingtalk.com,
  TOP 风格），**api.dingtalk.com 没有对应新版接口**；文档 2026-07 仍在持续维护，
  无下线公告。
  出处：<https://open.dingtalk.com/document/development/message-corpconversation-overview>
- 因此 `DingTalkApiClient` 「不接旧 topapi」的原决策（`api_client.py:21` 注释，
  原语境是审批流已有新版 API）需要为工作通知开**文档化例外**：新增
  `DINGTALK_OAPI_BASE_URL = "https://oapi.dingtalk.com"` 常量与
  `?access_token=` 查询参数的请求分支，例外范围仅限本篇 §2 的三个端点 + 发送端点。

### 1.1 发送接口

`POST https://oapi.dingtalk.com/topapi/message/corpconversation/asyncsend_v2?access_token=...`
出处：<https://open.dingtalk.com/document/development/asynchronous-sending-of-enterprise-session-messages>

请求体（本设计用到的子集）：

| 参数 | 类型 | 约束 |
|---|---|---|
| `agent_id` | Long | 必填；开发者后台应用详情页的 AgentID，由目标 App workspace 的版本化通知通道配置 |
| `userid_list` | String | 逗号分隔，**≤100 个**（本设计的批大小依据） |
| `msg` | JSON | 必填，**整体 ≤2048 字节**；单次仅一种 msgtype |

返回：`errcode/errmsg/request_id` + **`task_id`（Number）**——查进度/结果/撤回的句柄。
官方明示：**「接口发送成功不代表用户收到」**，被流控的用户静默丢弃。

不使用的参数（记录为非目标）：`dept_id_list`（≤20 部门）、`to_all_user`
——本设计的收件人语义是显式个人列表，部门/全员群发不开放给下游（防误伤 + 保住
getsendresult 能力，见 §2.2）。

## 2. 投递结果查询

### 2.1 getsendprogress

`POST .../topapi/message/corpconversation/getsendprogress`，参数
`agent_id + task_id`；返回 `progress.status`（0 未开始 / 1 处理中 / **2 处理完毕**）
与 `progress_in_percent`。**只能查 7 天内的任务**。
出处：<https://open.dingtalk.com/document/development/obtain-the-sending-progress-of-asynchronous-sending-of-enterprise-session>

### 2.2 getsendresult

`POST .../topapi/message/corpconversation/getsendresult`，参数 `agent_id + task_id`。
出处：<https://open.dingtalk.com/document/development/gets-the-result-of-sending-messages-asynchronously-to-the-enterprise>

两条硬限制（直接塑造了第 3 篇的管道设计）：

1. **只能查 24 小时内**的任务 → 对账窗口 24h；超窗后保持 `sent`，不推断 delivered；
2. **接收人 >100 的任务不支持本接口**（会超时）→ 发送必须 ≤100/批。

返回 `send_result` 名单与本设计状态映射：

| 名单字段 | 含义 | 映射 |
|---|---|---|
| `invalid_user_id_list` | 无效 userid | `failed(DINGTALK_REJECTED)` |
| `failed_user_id_list` | 发送失败 | `failed(DINGTALK_REJECTED)` |
| `forbidden_user_id_list` + `forbidden_list[{code,count,userid}]` | 被流控静默过滤 | code **143106** → `failed(DINGTALK_DUPLICATE)`；code **143105** → `failed(DINGTALK_DAILY_LIMIT)` |
| `read_user_id_list` / `unread_user_id_list` | 钉钉明确的回执分类 | `delivered`；不表示消费方可依赖的已读/合规事实 |

### 2.3 撤回接口（存在，本期非目标）

`POST .../topapi/message/corpconversation/recall`，参数 `agent_id + msg_task_id`，
**仅支持撤回 24h 内**。记录备用，不进本期契约。
出处：<https://open.dingtalk.com/document/development/notification-of-work-withdrawal>

## 3. 消息类型选择

msgtype 全集：text/image/voice/file/link/oa/markdown/action_card。
本设计开放三种（契约 `template` 取值），出处：
<https://open.dingtalk.com/document/development/message-types-and-data-format>

| template | 钉钉 msg 结构 | 官方字段约束 |
|---|---|---|
| `text` | `{"msgtype":"text","text":{"content":...}}` | content 建议 ≤500 字符 |
| `markdown` | `{"msgtype":"markdown","markdown":{"title":...,"text":...}}` | title 必填（首屏透出文案）；text 标注 ≤5000 字符，但受 msg 整体 2048 字节约束，**以先触发者为准** |
| `action_card`（整体跳转单按钮） | `{"msgtype":"action_card","action_card":{"title":...,"markdown":...,"single_title":...,"single_url":...}}` | `single_title` ≤20 字符；`single_url` ≤500 字符；markdown 建议 ≤1000 字符 |

- **不开放独立跳转（多按钮）action_card**：场景不需要，且 `btn_json_list` 的
  action_url 会诱导把操作语义塞进通知（应走应用内深链）。
- **2048 字节是所有模板共同的硬顶**：API 受理时把组装后的 msg JSON 按 UTF-8
  字节数校验（纯中文正文经验值约 550~600 字），超限 `422 VALIDATION_ERROR`
  ——比让钉钉拒收更早暴露问题。

### 3.1 深链打开行为（deeplink_url）

出处：<https://open.dingtalk.com/document/development/message-link-description>、
<https://open.dingtalk.com/document/development/link-jump-specification>

- 直接填 https 链接：移动端在钉钉内置浏览器以 H5 打开；**PC 端以系统默认浏览器打开**。
- 若要 PC 端侧边栏内打开需包一层跳转协议：
  `dingtalk://dingtalkclient/page/link?url={urlencode(https链接)}&pc_slide=true`。
- `dtmd://` 协议是机器人单聊里「点击代发消息」的交互协议，**不是** action_card
  跳转方式，不适用。
- **本设计结论**：契约的 `deeplink_url` 要求裸 https 链接；EasyAuth 侧不自动包
  `dingtalk://` 协议（EasyProject 是标准 Web 应用，浏览器打开是合理默认）。若下游
  future 需要 pc_slide 行为，自行传入 `dingtalk://dingtalkclient/page/link?...` 形式
  即可通过（校验放行 `https://` 与 `dingtalk://dingtalkclient/page/link?` 两种前缀，
  后者内嵌 url 参数仍须 https）。

## 4. 官方频控规则汇总

### 4.1 工作通知业务级（出处均为 §1.1 发送接口文档「频率限制」节）

官方警示原文：「**超出以下限制次数后，接口返回成功，但用户无法接收到**」。

| 限制 | 数字 | 对本设计的意义 |
|---|---|---|
| 单次任务接收人 | 企业内部应用 ≤5000 人 | 契约单请求收件人上限 500，远低于此 |
| `userid_list` | ≤100 | 批大小 = 100 |
| 相同内容对同一用户 | **一天仅 1 次**（143106） | ① 吸收 at-least-once 崩溃重发；② 两条不同消息内容完全相同时第二条会静默失败——契约提示下游在 content 中带业务变量（如任务名/日期） |
| 单应用对单人日上限 | 企业内部应用 **500 条/天**（143105） | 正常量级碰不到；碰到即数据回执标 failed |
| 接收速率 | 每分钟最多 5000 人次收到（143103/143104 QPM） | send 调用报错型频控 → throttled + 120s 退避（第 3 篇 §4） |

### 4.2 平台通用频控

出处：<https://open.dingtalk.com/document/development/how-to-process-api-throttling-on-the-dingtalk-server>

| 维度 | 规则 | 错误码 |
|---|---|---|
| 应用×接口 QPS | ≤20/s | 90018 |
| IP | 20 秒内 1 万次触发 5 分钟封禁 | 无（返回 HTML） |
| 全局接口 QPS | 有阈值未公布 | 90002 |
| **月调用量** | **标准版（免费版）组织全部企业内部应用共 5000 次/月**；专业版/专属版可扩容 | 90020 |

关键核查：《不纳入调用量限制的接口清单》
（<https://open.dingtalk.com/document/development/basic-interfaces-such-as-log-off-and-address-book>）
**不含** corpconversation 系列——工作通知计入 5000 次/月额度；获取 accessToken
在豁免清单内不计。**上线前必须确认本企业钉钉版本**：若为标准版，按「send+对账×2」
的调用系数，月安全预算约 1600 条消息，需要么升级专业版，要么关闭对账开关
（第 3 篇 §5）并压缩配额。此项列入第 7 篇发布前检查单。

## 5. 凭据结论：每 App 独立、版本化通知通道

- **发送工作通知不需要额外权限点**：官方明示「应用创建后，系统默认添加身份验证和
  **消息通知**接口权限」，asyncsend_v2 及查询/撤回接口的权限要求均为「调用企业 API
  的基本权限」。
  出处：<https://open.dingtalk.com/document/development/add-api-permission>
- **通知展示身份按业务 App 隔离**：App owner 在 App workspace 配置
  `dingtalk_app_key/app_secret/agent_id`。每次更新创建新通道版本，
  每个 App 最多一条 active 通道；不使用全局 `IntegrationSettings` 作为 notify 发送身份。
- **消息冻结通道**：`202` 受理时固定通道版本，后续更换 active 通道不影响已受理消息。
- **access_token 复用现有获取与缓存逻辑**：客户端使用当前通道的 app key/secret 走
  `POST https://api.dingtalk.com/v1.0/oauth2/accessToken`（7200s 有效期、提前 120s
  刷新缓存，`api_client.py:92-128`）。新版 token 与旧版 oapi 接口为同一凭证体系、
  可互用（官方未在单页逐字声明，社区广泛验证；旧版 gettoken 文档
  <https://open.dingtalk.com/document/development/obtain-orgapp-token>，新版
  <https://open.dingtalk.com/document/development/obtain-the-access-token-of-an-internal-app>）。
  当前实现按该路径获取 token，没有旧版 gettoken 兼容分支。

## 6. 对设计的六条工程结论（供快速引用）

1. 发送走 asyncsend_v2（旧版，唯一通道）；**受理成功 ≠ 送达**，被流控者静默丢弃；
2. 批大小 100（userid_list 上限，同时保住 getsendresult 能力）；回执窗口 24h、
   进度窗口 7 天；
3. 「相同内容同人一天 1 次」的服务端去重：既是崩溃重发的天然吸收层，也是下游必须
   知道的内容设计约束（写入契约与集成指南）；
4. send 调用报错型频控只有 QPS/QPM（分钟级），退避 120s 足够；日额度/重复内容是
   收件人级静默失败，只能对账发现，不可重试；
5. msg 整体 ≤2048 字节是最先触发的内容上限，API 受理时前置校验；
6. 通知凭据按 App 隔离：owner 配置版本化通道，受理消息冻结通道版本。
