## 1. 设计稿内部矛盾与裁决

### 1.1 `deeplink_title` 缺列（第 1 篇 §N2 ↔ 第 2 篇 §2 ↔ 第 6 篇 §1）

- **矛盾**：契约 §N2 定义可选 `deeplink_title`（≤20 字符，缺省「查看详情」），§X N-6
  记为对草案的刻意新增；但第 2 篇 NotifyMessage 字段清单漏列，payload_hash 规范化
  口径也未纳入；第 6 篇 `send_notification` 签名同样遗漏（与其自身「方法与契约
  一一对应」的声明矛盾）。
- **裁决**：契约为准。PR-4 补 `NotifyMessage.deeplink_title`（max_length=20, blank，
  迁移 `notify/0002`）、受理持久化、`compute_payload_hash` 纳入、投递重建从列读取；
  PR-5 SDK `send_notification` 补可选参数。当时 `accept_notify_message` 无任何调用方、
  无存量行，属零成本窗口，无历史 hash 兼容问题。
- **影响**：第 2 篇 §2 字段表、§2 hash 口径与第 6 篇 §1 签名应按此更新（本篇即为记录，
  原文未改）。

### 1.2 频控退避 120s vs 1h（第 3 篇 §4 ↔ 第 5 篇 §5）

- **矛盾**：第 5 篇 §5 行文写「命中钉钉限流按 throttled + 1h 退避」，但其括号出处
  指向第 3 篇 §4；第 3 篇明文 `NOTIFY_THROTTLE_RETRY_SECONDS = 120` 并有论证
  （QPS/QPM 皆分钟级窗口）；第 1 篇 §N3 亦写 120s。
- **裁决**：两篇权威文（01 契约、03 管道算法）一致取 **120s**；第 5 篇「1h」判定为
  引用笔误，不构成独立要求。

### 1.3 凭据缺失不判终态（第 3 篇 §8 语义延伸）

- **背景**：`DingTalkNotConfiguredError` / agent_id 空串在投递时如何处置，设计未明文。
- **裁决**：视为可运维修复项（对齐第 3 篇 §8 依赖健康检查的定位），保持 pending +
  常规 4 段退避，配置补齐后自动续投；重试耗尽仍由 `EXHAUSTED` 兜底。不判
  `DINGTALK_REJECTED`（该码契约语义是「钉钉回执」，凭据缺失并非回执）。

## 2. 有据实现偏差（不改变契约语义）

- **D1 示例行序**（第 1 篇 §D）：原示例与「name 升序」规则矛盾；
  实现与 `tests/contract_samples/directory/users_list.json` 始终按规则排序，正式示例已同步修正。
- **list 类聚合审计**（第 5 篇 §4）：按设计「cache 计数 + 小时翻转落库」实现；固有属性：
  末次调用后 TTL(3700s) 内无新请求，则尾部小时计数不落库。审核裁定为该机制固有、
  可接受；如需强保证需另加定时 flush（设计变更，未做）。
- **SDK capabilities 重复值**（第 6 篇 §4）：早期实现曾 fail-fast 拒绝重复；
  SDK 0.3.0 已与服务端对齐，按 trim 后值去重校验、接受未知非空字符串，
  同时保持 `validate_manifest` 原样返回输入对象。
- **console 能力开关 API 路由形状**（第 8 篇 PR-1，设计未写死）：实现为
  `GET /console/api/v1/apps/{app_key}/capabilities` + per-capability `GET|PUT`，
  风格对齐 `managed_scope_policy_api`；仅 enabled 翻转写审计。
- **oapi 业务 errcode**：`DingTalkApiRequestError` 增加独立 `errcode` 字段，不再复用
  `status_code`（第 4 篇未规定异常载体字段，属实现级修正）。
- **GET 通知状态遇能力关闭也记 `app_notify_rejected` 审计**：较第 5 篇 §4 的「POST 被拒」
  略宽，无害，保留。

## 4. EasyTrade 接入前的收敛实施

- `58aa3f9`：目录条目增加 `email`、`mobile`、`employee_number`、`status`，
  保留 `active`；从权威快照消失的员工改为 `departed` tombstone，不物理删除。
  所有目录响应增加多企业 `directory_snapshot`，列表支持 `snapshot_id` 固定分页。
- `2d5a8e7`：`stale` 以 EasyAuth 本地成功事务提交时间判定；上游
  `snapshot_at` 仅作展示元数据，并暴露 `snapshot_at_status`。
- `e7b0860`、`6a7314c`：平台能力改为 App capability 与 credential capability
  双层逻辑与。超管开通 App 能力，App owner 授权具体凭据；
  manifest 声明不自动开通或授权。`0025_credential_capabilities` 为兼容存量调用，
  会将已启用 App capabilities 回填给该 App 全部 active 静态/OAuth 凭据；
  上线后必须审计、拆分凭据并撤销旧 permission token 的多余 capability。
- `aedac67`、`1bbb548`：通知通道改为每 App 独立、版本化配置，由 App owner 在
  workspace 维护。消息受理时冻结非空 channel；历史消息迁移安全回填，
  App 缺少 active 完整通道时 fail-fast 阻断迁移。回执没有明确
  read/unread/失败名单时保持 `sent`，超过 24h 也不乐观推断 `delivered`。
  `delivered` 仅是钉钉明确 send-result 回执分类，不是已读、审批知悉或法务送达证据。
- `5643b20`、`4d90589`、`ebed87d`、`8f04b9e`：SDK 升级为 `0.3.0`，结构化暴露
  `error_code`、`details`、`Retry-After`、`retryable` 与 transport error，安全关闭
  HTTP error response，生产默认要求 HTTPS，目录搜索支持 `snapshot_id` 固定分页；
  SDK 本身不自动重试。
- `45eb2ac`、`9a81cb8`：目录用户、部门和主管条目增加 `source_slug`、`corp_id`
  与 opaque scoped `user_ref` / `department_ref`；详情、过滤和通知按 source/corp
  消歧。旧裸 user_id、`dt:`、原始部门 ID 仅作唯一匹配兼容，歧义为目录 HTTP 409
  或通知逐收件人 `USER_AMBIGUOUS`，畸形 scoped ref 为目录 HTTP 422。
- `36acc2e`、`1b12d4a`：通知通道绑定权威目录 source/corp，控制台只允许从
  sync state、用户镜像、部门镜像的作用域并集中选择；失效 scope 健康检查 unhealthy，
  越界收件人 `USER_SCOPE_MISMATCH`。对账增加持久化 `last_reconciled_at`，按最多 50 个
  唯一 channel/task pair 公平轮转；日配额由 App 行锁串行计数与写入。
- `555e114`、`7cb4f47`：App workspace 通知通道 UI 改为 validated scope 下拉，
  对历史 invalid scope 显示不可选修复态，owner 必须改选有效作用域后保存新版本；
  secret 首次必填、后续可复用且永不回显的规则保持不变。
