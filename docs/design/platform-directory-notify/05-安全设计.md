# 安全设计（目录字段暴露 / 通知目标范围 / 限流 / 审计）

> 第 5 篇。契约见第 1 篇；能力开关的数据模型见第 2 篇 §1。

## 1. 能力默认关闭：`AppCapability` 开关

- 任何 app 拿到 app token ≠ 能拉目录/发通知。两项能力都必须由超管在 console 为
  具体 app 显式开通（`AppCapability(capability, enabled=True)`），开关动作写审计。
- manifest 可新增 `capabilities: ["directory", "notify"]` 顶层申明（见第 6 篇 §4），
  但**申明只是意图表达**：用于 console 提示管理员「该 app 请求了哪些平台能力」，
  不自动开通。这与 MANAGED_USERS 的治理哲学一致——下游声明、平台裁决。
- 能力未开通时返回 `403 PERMISSION_DENIED`，消息明示「应用未开通目录能力/通知能力」，
  与「凭据无效 401」严格区分，避免下游误判成 token 问题。

## 2. 目录字段暴露策略（最小化评估结论)

逐字段裁决（镜像里有什么 → 给不给）：

| 字段 | 来源 | 结论 | 理由 |
|---|---|---|---|
| `user_id`(authentik_user_id) | `UserMirror` | ✅ 暴露（可空） | 权限 API 的既有对外标识（ADR-002 明确成员主键用它） |
| `dingtalk_user_id` | `DingTalkUserMirror.user_id` | ✅ 暴露 | 通知与@提及的必要标识；本身不含个人敏感内容 |
| `name` / `avatar_url` / `title` | 镜像 | ✅ 暴露 | 选人器展示的最低需求 |
| `departments`（id+名称） | `DingTalkUserMirror.department_ids` + 部门镜像 | ✅ 暴露 | 部门树选人的必要数据 |
| `active` | 镜像 status 派生 | ✅ 暴露 | 下游需要过滤离职/禁用 |
| 主管摘要（详情/manager 端点） | `manager_userid` 解析 | ✅ 暴露 | 升级提醒场景的刚需 |
| `email` | `UserMirror.email` | ❌ 不暴露 | 通知统一走 EasyAuth，下游没有自行触达用户的正当需求；邮箱外泄面 = 钓鱼/撞库面。未来确有需求再按 app 加字段分级开关 |
| `employee_number` | `UserMirror.employee_number` | ❌ 不暴露 | 工号是 HR 域标识，与授权/通知场景无关；跨系统关联риск高 |
| 手机号 | **未入镜像** | ❌ 不存在 | 镜像层从未同步手机号，天然合规；不为目录 API 引入 |
| `dingtalk_union_id` / `corp_id` | 镜像 | ❌ 不暴露 | 下游用不到；unionId 属跨应用身份标识，泄露面大于价值 |
| `manager_chain`（完整主管链） | `DingTalkUserOrgContext` | ❌ 暂不暴露 | 场景只要「直接主管」；整链暴露=组织结构测绘。留作未来带治理的扩展 |

**部门树也是组织情报**：departments 端点同样受 `directory` 能力开关约束，
不单独放开。

## 3. 通知目标范围策略：结论与权衡

**结论：允许通知任意 `active` 目录用户，不限制为「对该 app 持有 AccessGrant 的用户」。**

权衡过程：

| 方案 | 支持的场景 | 问题 |
|---|---|---|
| A. 仅 AccessGrant 持有者 | 常规成员通知 | **直接击穿两个既定场景**：① 逾期升级要通知负责人的主管——主管很可能没有 easyproject 的授权；② @提及/指派尚未开通该 app 的同事（通知恰恰是引导其登录开通的入口）。且引入「通知能力 ↔ 授权数据」的耦合：授权撤销瞬间连告别通知都发不出去 |
| B. 任意 active 目录用户（选定） | 全部既定场景 | 被攻陷/恶意的 app 可向全员发骚扰消息 |
| C. B + per-app 收件人白名单/范围配置 | 同 B | 治理成本高，第一个消费方就要求全员范围，白名单形同虚设 |

选 B 的风险缓解是**纵深防御而非范围限制**：

1. 能力开关默认关（§1）——能发通知的 app 是管理员逐个授信的；
2. 每 app 双层限流 + 每日收件人配额（§5）——把「骚扰全员」的爆炸半径限制在配额内；
3. 全量审计（§4）——谁在什么时候给谁发了什么类别的通知，可追溯；
4. 内容治理兜底：title/content 长度上限、deeplink_url 必须 https（防钉钉卡片
   变成钓鱼跳板）；
5. `active` 硬约束：disabled/departed 用户一律拒发（`USER_INACTIVE`），
   离职员工不再收到内部系统消息。

## 4. 审计点清单

沿用 `AuditService.record(AuditRecord(...))`（`audit/services.py:24`），
`actor_type="app", actor_id=app_key`（app 发起）或 `"system"`（管道内部），
action 常量化：

| action | 触发点 | target / metadata 要点 |
|---|---|---|
| `app_capability_enabled` / `app_capability_disabled` | console 开关能力 | target=app；metadata: capability、updated_by |
| `app_directory_queried` | 目录 list/detail/manager/subordinates/departments 每次调用 | target_type=`directory`; metadata: endpoint、q 是否非空（**不记 q 原文**，检索词可能含人名隐私）、结果数、credential_id |
| `app_notify_accepted` | POST 受理成功（含幂等命中） | target=message_id; metadata: template、biz_tag、recipient_total、dedup 命中与否、credential_id |
| `app_notify_rejected` | POST 被拒（配额/校验/能力关闭） | metadata: error_code、recipient_count |
| `notify_delivered` | 消息到达终态 | target=message_id; metadata: 状态、sent/failed 计数、attempts |
| `notify_delivery_exhausted` | 重试耗尽 | 同 webhook_delivery_exhausted 口径，伴随 logger.error 供告警 |

**目录查询审计的采样口径**：选人器是高频交互（每键一查经下游后端转发），逐次审计
会让 `AuditLog` 膨胀。折衷：list 搜索类审计按「app × 端点 × 小时」聚合记一条
（metadata 里累计次数，实现为 cache 计数 + 小时翻转落库）；detail/manager/
subordinates 与全部 notify 事件仍逐次记。

## 5. 限流设计（全部复用 `config/rate_limit.py` 固定窗口原语）

沿用现有三视图的双层模式（认证失败按 IP，业务按 credential/app）：

| namespace | 维度 | 阈值（默认） | 说明 |
|---|---|---|---|
| `directory-authfail` / `notify-authfail` | IP | 30 次 / 300s | 与 `perm-query-authfail` 同档 |
| `directory-rate` | credential_id | 240 次 / 60s | 与权限查询同档；覆盖选人器击键频率（下游应做防抖） |
| `notify-post-rate` | app_id | `rate_per_minute`（默认 60）/ 60s | 写操作对齐审批创建档；按 app 而非 credential——多凭据不应放大配额 |
| `notify-daily-quota` | app_id | `daily_recipient_quota`（默认 5000 收件人·次/自然日） | DB count 口径（第 2 篇 §3.2），超限 429 + `Retry-After` 到次日零点 |
| `notify-status-rate` | credential_id | 240 次 / 60s | GET 状态查询 |

- 钉钉侧全局频控（对整个企业应用生效，第 4 篇 §4）是另一层：投递任务命中钉钉限流
  时按 throttled 状态 + 1h 退避处理（第 3 篇 §4），**不反压到 API 层**——下游的 202
  始终成立，靠状态查询看结果。
- 所有 429 响应带 `Retry-After` 头（对齐 `approval_views.py:377` 的做法）。

## 6. 其余安全要点

- **认证复用不新造**：目录与通知端点使用与现有公共 API 完全相同的
  `authenticate_permission_query_token` → `AppPrincipal` → 「路径 app_key 必须等于
  principal.app_key，否则 403」三段式（`api/views.py:53` 同款）。凭据体系零改动。
- **`dt:` 引用不构成越权面**：通知收件人允许 `dt:<钉钉userid>` 引用（契约 §N1），
  解析仅在镜像表内进行，不回源钉钉/Authentik；引用不存在 → 该收件人
  `failed(USER_NOT_FOUND)`，不泄露「该 userid 是否存在于企业」之外的信息——
  这与目录能力本身的可见范围一致（开了目录能力本来就能枚举 active 用户）。
  仅开通 notify 未开通 directory 的 app 理论上可借失败码探测 userid 存在性，
  接受此风险：两能力的授信主体同为内部应用，且探测行为受 `notify-post-rate` 限速
  并全量落审计。
- **deeplink_url 校验**：必须 `https://`；不做域名白名单（下游深链域名多样，
  白名单维护成本 > 收益；卡片仅在企业内部钉钉可见，且发起方是授信 app）。
- **内容不落审计原文**：`AuditLog.metadata` 只记 template/biz_tag/计数，
  不记 title/content——通知正文可能含业务敏感信息（如审批金额），审计目标是
  「谁发了、发给谁、多大量」，不是留存内容副本；内容本体在 `NotifyMessage` 表内
  受 180 天保留期约束。
- **无 SSRF 面**：通知出站目标是常量域名（`oapi.dingtalk.com`/`api.dingtalk.com`），
  不接受调用方提供的 URL 作为请求目标（deeplink_url 只作为消息载荷透传给钉钉，
  EasyAuth 从不请求它）。
