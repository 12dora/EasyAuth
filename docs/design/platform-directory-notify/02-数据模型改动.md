# 数据模型改动（公共用户目录 API + 统一钉钉通知 API）

> 本文是《平台能力升级：公共用户目录 + 统一钉钉通知》设计的第 2 篇。API 契约见
> `01-API契约-权威版.md`，投递管道见 `03-Celery任务与投递管道.md`。

## 0. 总原则

- **目录能力零新表**：目录数据完全复用 `accounts` 现有镜像表
  （`DingTalkUserMirror` / `DingTalkDepartmentMirror` / `DingTalkUserOrgContext` /
  `UserMirror`，见 `src/easyauth/accounts/models.py`），只补两个查询索引。
- **通知能力两张新表**：`NotifyMessage` + `NotifyRecipient`，落在新 Django app
  `easyauth.notify` 下（归属论证见第 7 篇实施计划 §1）。
- **能力开关一张新表**：`AppCapability`，落在 `applications` app（它是 App 维度的平台配置，
  与 `AppCredential`、`AppWebhookConfig` 同类）。
- 字段风格对齐仓库惯例：显式类型标注的 `models.CharField[str, str]`、`Final` 常量枚举 +
  `choices`、`ClassVar` 约束列表、审计型表禁物理删除（参照 `UserMirror.delete` 与
  `AuditLog` 的做法）。

## 1. `applications.AppCapability` —— per-app 平台能力开关

现有 `App` 模型（`applications/models.py:82`）只有
`app_key/name/description/is_active/catalog_version`，**没有任何 capability 挂点**。
目录与通知都属于「默认关闭、管理员显式授予」的平台能力（安全论证见第 5 篇 §1），
需要新表承载：

```python
# applications/models.py 追加

CAPABILITY_DIRECTORY: Final = "directory"
CAPABILITY_NOTIFY: Final = "notify"
CAPABILITY_CHOICES: Final[tuple[tuple[str, str], ...]] = (
    (CAPABILITY_DIRECTORY, "directory"),
    (CAPABILITY_NOTIFY, "notify"),
)


class AppCapability(models.Model):
    # App 维度的平台能力开关(目录/通知)。manifest 只能"申明"能力,
    # 开通必须由超管在 console 显式执行(申明 ≠ 授予)。
    app: models.ForeignKey[App, App] = models.ForeignKey(
        App,
        on_delete=models.CASCADE,
        related_name="platform_capabilities",
    )
    capability: models.CharField[str, str] = models.CharField(
        max_length=32,
        choices=CAPABILITY_CHOICES,
    )
    enabled: models.BooleanField[bool, bool] = models.BooleanField(default=False)
    # 能力级配置。notify: {"daily_recipient_quota": 5000, "rate_per_minute": 60};
    # directory: 预留(如未来的字段分级)。空 dict 表示全部取默认值。
    config: models.JSONField[dict[str, JsonValue], dict[str, JsonValue]] = models.JSONField(
        default=dict,
        blank=True,
    )
    updated_by: models.CharField[str, str] = models.CharField(max_length=128, blank=True)
    created_at / updated_at  # auto_now_add / auto_now, 同仓库惯例

    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=["app", "capability"],
                name="applications_app_capability_unique",
            ),
        ]
```

**读取口径**（服务层函数，供 API 视图调用）：

```python
def app_capability_enabled(app_id: int, capability: str) -> bool
def app_capability_config(app_id: int, capability: str) -> dict[str, JsonValue]
```

- 无行或 `enabled=False` → 能力关闭，公共 API 返回 `403 PERMISSION_DENIED`
  （错误消息区分「应用未开通目录能力」/「应用未开通通知能力」，方便下游排障）。
- `config` 中 notify 支持两个键（缺省时取 settings 级默认）：
  - `daily_recipient_quota`：每自然日（Asia/Shanghai）收件人·次配额，默认
    `EASYAUTH_NOTIFY_DEFAULT_DAILY_RECIPIENT_QUOTA = 5000`；
  - `rate_per_minute`：每分钟 POST 次数，默认 `EASYAUTH_NOTIFY_DEFAULT_RATE_PER_MINUTE = 60`。

**console 管理面**：`admin_console` 增加能力开关 API（superuser-only，风格对齐
`webhook_config_api.py`），开/关均写审计（`app_capability_enabled` /
`app_capability_disabled`，见第 5 篇 §4 审计点清单）。

## 2. `notify.NotifyMessage` —— 通知消息主表

一次 `POST /notify/messages` 调用 = 一行。设计原型是 `workflows.ApprovalInstance`
（UUID 公开主键 + `biz_key` 幂等 + `payload_hash` 冲突检测，`workflows/models.py:144`）
与 `webhooks.WebhookDelivery`（claim/lease/attempts 投递骨架，`webhooks/models.py:105`）
的合体：**幂等语义学审批实例，投递语义学 webhook 投递**。

```python
# notify/models.py

NOTIFY_TEMPLATE_TEXT: Final = "text"
NOTIFY_TEMPLATE_MARKDOWN: Final = "markdown"
NOTIFY_TEMPLATE_ACTION_CARD: Final = "action_card"

NOTIFY_MESSAGE_STATUS_PENDING: Final = "pending"            # 已受理, 未开始投递
NOTIFY_MESSAGE_STATUS_SENDING: Final = "sending"            # 投递中(含重试间隙)
NOTIFY_MESSAGE_STATUS_COMPLETED: Final = "completed"        # 全部收件人 sent/delivered
NOTIFY_MESSAGE_STATUS_PARTIALLY_FAILED: Final = "partially_failed"
NOTIFY_MESSAGE_STATUS_FAILED: Final = "failed"              # 全部收件人失败


class NotifyMessage(models.Model):
    # 公开 message_id 即 UUID 主键(对齐 ApprovalInstance.id 的做法, 不泄漏自增序号)。
    id: models.UUIDField = models.UUIDField(primary_key=True, default=uuid.uuid4, editable=False)
    app: models.ForeignKey[App, App] = models.ForeignKey(
        App, on_delete=models.CASCADE, related_name="notify_messages",
    )
    template: models.CharField  # max_length=16, choices=上述三种
    title: models.CharField     # max_length=100, blank=True(text 模板可空)
    content: models.TextField   # API 层校验: 组装后的钉钉 msg JSON ≤ 2048 字节(官方硬限, 见第 4 篇 §3)
    deeplink_url: models.CharField  # max_length=512, blank=True; action_card 必填(API 层校验)
    # APP 内幂等键: 同 (app, dedup_key) 只受理一次, 永久有效(非 24h 窗口, 论证见 §2.1)。
    dedup_key: models.CharField     # max_length=128, blank=True
    # 规范化载荷(template/title/content/deeplink_url/recipients 排序后)的 sha256;
    # dedup_key 命中但 payload_hash 不同 → 409 CONFLICT(同 ApprovalInstance.payload_hash 语义)。
    payload_hash: models.CharField  # max_length=64
    biz_tag: models.CharField       # max_length=64, blank=True, db_index=True
    status: models.CharField        # max_length=20, choices=上述五种, default=pending

    # ---- 投递骨架(照抄 WebhookDelivery 的 claim/lease/attempts 三件套) ----
    attempts: models.PositiveIntegerField   # default=0, 消息级投递轮次
    claim_token: models.CharField           # max_length=32, blank=True
    lease_expires_at: models.DateTimeField  # null=True, blank=True
    last_error: models.TextField            # blank=True

    # ---- 去范式化计数(收件人状态汇总, 投递任务每轮结束时刷新) ----
    recipient_total: models.PositiveIntegerField    # default=0
    recipient_sent: models.PositiveIntegerField     # default=0, 含 delivered
    recipient_failed: models.PositiveIntegerField   # default=0

    # ---- 调用方溯源(审计口径对齐 AuditLog.metadata 里的 credential 字段) ----
    requested_credential_type: models.CharField  # max_length=32 (static_token|oauth_client)
    requested_credential_id: models.IntegerField # AppCredential/OAuthClientBinding 主键

    completed_at: models.DateTimeField  # null=True, 终态时间
    created_at / updated_at

    class Meta:
        constraints = [
            # 幂等: 空 dedup_key 不参与唯一约束
            models.UniqueConstraint(
                fields=["app", "dedup_key"],
                condition=~Q(dedup_key=""),
                name="notify_message_dedup_unique",
            ),
            models.CheckConstraint(  # 状态白名单, 同 WebhookDelivery 的做法
                condition=Q(status__in=[...5 个状态...]),
                name="notify_message_status_supported",
            ),
        ]
        indexes = [
            models.Index(fields=["app", "status"], name="notify_msg_app_status_idx"),
            models.Index(fields=["app", "created_at"], name="notify_msg_app_created_idx"),
            models.Index(fields=["status", "lease_expires_at"], name="notify_msg_lease_idx"),
        ]
        ordering = ["-created_at", "id"]
```

### 2.1 为什么 `dedup_key` 是永久唯一而不是「24h 窗口」

- 草案只说「同一事件重复触发要可去重」，没有规定窗口。窗口去重需要 TTL 存储
  （cache 会被清空、重启即失忆），DB 部分唯一索引则是**零额外设施的强幂等**，
  且与仓库现成的 `ApprovalInstance(app, template, biz_key)` 幂等语义一致。
- 语义约定写进契约：**dedup_key 代表「事件发生」的身份**，调用方应把事件的
  发生标识编进 key（如 `task-assigned:123:v2`、`overdue-remind:123:2026-07-16`）；
  想「明天再发一遍」就换 key。这比服务端猜测窗口更可控。
- 代价：`notify_message` 行永久保留 dedup_key。配套第 3 篇 §6 的保留期清理任务会
  在清理老消息时一并释放 key（清理期 ≥ 180 天，远超任何合理的重触发窗口）。

### 2.2 消息聚合状态机

```
pending ──(投递任务首轮启动)──▶ sending ──▶ completed          (failed_count == 0)
                                   │──▶ partially_failed       (0 < failed < total)
                                   │──▶ failed                 (failed == total)
   ▲                               │
   └──(重试排程, 不改状态)◀────────┘   重试期间保持 sending
```

终态判定发生在「无 pending/throttled 收件人」或「重试耗尽」时（见第 3 篇 §3）。

## 3. `notify.NotifyRecipient` —— 收件人投递记录

一个收件人 = 一行；投递状态机的最小粒度。

```python
NOTIFY_RECIPIENT_STATUS_PENDING: Final = "pending"
NOTIFY_RECIPIENT_STATUS_SENT: Final = "sent"          # 钉钉 asyncsend 受理(拿到 task_id)
NOTIFY_RECIPIENT_STATUS_DELIVERED: Final = "delivered"  # 回执确认送达(getsendresult 对账)
NOTIFY_RECIPIENT_STATUS_FAILED: Final = "failed"      # 终态失败(error_code 注明原因)
NOTIFY_RECIPIENT_STATUS_THROTTLED: Final = "throttled"  # 钉钉频控拒收, 等待重试

# error_code 枚举(契约 §N4 对外暴露同名值)
NOTIFY_ERROR_USER_NOT_FOUND: Final = "USER_NOT_FOUND"        # 引用无法解析到镜像用户
NOTIFY_ERROR_NO_DINGTALK_ID: Final = "NO_DINGTALK_ID"        # UserMirror 无钉钉绑定
NOTIFY_ERROR_USER_INACTIVE: Final = "USER_INACTIVE"          # 目录状态非 active(禁用/离职)
NOTIFY_ERROR_DINGTALK_REJECTED: Final = "DINGTALK_REJECTED"  # 钉钉回执: 无效 userid 或发送失败
NOTIFY_ERROR_DINGTALK_DUPLICATE: Final = "DINGTALK_DUPLICATE"    # 钉钉流控 143106: 相同内容同人一天仅 1 次
NOTIFY_ERROR_DINGTALK_DAILY_LIMIT: Final = "DINGTALK_DAILY_LIMIT"  # 钉钉流控 143105: 单应用对单人日上限(500 条)
NOTIFY_ERROR_EXHAUSTED: Final = "EXHAUSTED"                  # 重试耗尽仍未送达


class NotifyRecipient(models.Model):
    message: models.ForeignKey[NotifyMessage, NotifyMessage] = models.ForeignKey(
        NotifyMessage, on_delete=models.CASCADE, related_name="recipients",
    )
    # 调用方原始引用(原样回显): "authentik-uid" 或 "dt:钉钉userid"
    raw_ref: models.CharField        # max_length=200
    # 解析结果: 绑定的 UserMirror(可空: dt: 引用可能没有登录过的镜像行)
    user: models.ForeignKey[UserMirror | None, UserMirror | None]  # null=True, on_delete=PROTECT
    dingtalk_corp_id: models.CharField   # max_length=128, blank=True
    dingtalk_userid: models.CharField    # max_length=128, blank=True("" = 解析失败)
    status: models.CharField             # max_length=16, choices=上述五种, default=pending
    error_code: models.CharField         # max_length=64, blank=True
    error: models.TextField              # blank=True, 人读明细(如钉钉 errmsg)
    # 钉钉 asyncsend_v2 返回的 task_id(int64, 存字符串防精度问题); 同一批次的收件人共享。
    dingtalk_task_id: models.CharField   # max_length=64, blank=True, db_index=True
    sent_at: models.DateTimeField        # null=True
    delivered_at: models.DateTimeField   # null=True
    created_at / updated_at

    class Meta:
        constraints = [
            # 解析后按钉钉 userid 去重: 同一消息同一真人只投一次
            # (user_id 引用与 dt: 引用指向同一人时, 第二个引用在受理时即被合并)。
            models.UniqueConstraint(
                fields=["message", "dingtalk_userid"],
                condition=~Q(dingtalk_userid=""),
                name="notify_recipient_target_unique",
            ),
            models.CheckConstraint(
                condition=Q(status__in=[...5 个状态...]),
                name="notify_recipient_status_supported",
            ),
        ]
        indexes = [
            models.Index(fields=["message", "status"], name="notify_rcpt_msg_status_idx"),
        ]
        ordering = ["message_id", "id"]
```

### 3.1 收件人状态机

```
            解析失败(受理时即判)
pending ────────────────────────▶ failed (USER_NOT_FOUND / NO_DINGTALK_ID / USER_INACTIVE)
   │
   │ asyncsend 受理成功
   ├──▶ sent ──(getsendresult: read/unread 名单或超期未见失败)──▶ delivered
   │      ├──(getsendresult: invalid/failed_user_id_list)──▶ failed (DINGTALK_REJECTED)
   │      └──(getsendresult: forbidden_list, code 143106/143105)──▶ failed
   │              (DINGTALK_DUPLICATE / DINGTALK_DAILY_LIMIT — 钉钉侧静默丢弃, 只能对账发现)
   │
   │ asyncsend 调用被频控(QPM/QPS 类 errcode, 表见第 4 篇 §4)
   ├──▶ throttled ──(重试轮次)──▶ pending(逻辑上重新参与下一轮) / failed (EXHAUSTED)
   │
   │ 消息级重试耗尽
   └──▶ failed (EXHAUSTED)
```

**注意**：`throttled → 重试` 不物理改回 pending，投递任务每轮处理
`status IN (pending, throttled)` 的收件人（见第 3 篇 §3），状态字段忠实记录最近一次结果。

### 3.2 每日配额的计数口径

**不建配额计数表**。每日已用量 = 当日 `NotifyRecipient` 行数：

```python
NotifyRecipient.objects.filter(
    message__app_id=app_id,
    created_at__gte=当日零点(Asia/Shanghai),
).count()
```

- 受理时先查后写（在事务内），配额超限拒绝整个请求（`429 THROTTLED`，见契约 §N2）。
- 依据：内部通知量级（数千/日）下这个 count 走 `notify_msg_app_created_idx` +
  message join 足够快；省一张需要并发 upsert 的计数表。若未来量级上升，
  再引入 `F()` 自增的日计数表，口径不变。

## 4. `accounts` 既有表：只加索引，不加字段

目录 API 的查询模式与现有索引的差距：

| 查询 | 现状 | 改动 |
|---|---|---|
| 选人器搜索 `DingTalkUserMirror.filter(status="active", name__icontains=q)` | 无索引，全表扫 | **不加索引**。目录量级（几百~几千行）顺序扫描 < 1ms；pg_trgm 留作未来优化（写进第 7 篇遗留项） |
| 直接下属 `DingTalkUserMirror.filter(manager_userid=x)` | 无索引 | 新增 `models.Index(fields=["source_slug", "corp_id", "manager_userid"], name="accounts_dt_user_manager_idx")` |
| 目录行 → 登录绑定 `UserMirror.filter(dingtalk_corp_id=c, dingtalk_userid=u)` | 无索引（目录同步的逐行回填也在用这个查询，`directory_sync.py:437,477`） | 新增 `models.Index(fields=["dingtalk_corp_id", "dingtalk_userid"], name="accounts_user_dingtalk_idx")`——同时让每 300s 的同步回填受益 |
| 部门列表 `DingTalkDepartmentMirror.filter(parent_id=p)` | 无索引 | 不加。部门表更小，扫描可忽略 |

两个索引都是纯 `AddIndex` 迁移，无数据回填、可在线执行、可无损回滚。

### 4.1 明确不做的字段改动

- **不给 `DingTalkUserMirror` 加 pinyin 列**：镜像上游（Authentik 钉钉目录端点）不提供
  拼音；自造拼音列需引入转换依赖并处理多音字。契约把拼音检索定义为消费方职责
  （详见契约 §D1 与差异对照表 D-6）。
- **不给 `UserMirror` 加 mobile 列**：手机号从未入镜像，目录 API 字段最小化评估
  （第 5 篇 §2）也不允许暴露，无需引入。
- **不建目录物化视图/缓存表**：目录读模型 = 镜像表实时查询 + HTTP 缓存头
  （数据本身最多滞后一个同步周期 300s，缓存约定见契约 §D 开头）。

## 5. 迁移清单与执行顺序

| # | app | 迁移内容 | 风险 |
|---|---|---|---|
| 1 | `applications` | `CreateModel AppCapability` + unique 约束 | 新表，无风险 |
| 2 | `notify`（新 app） | `CreateModel NotifyMessage` / `NotifyRecipient` + 全部约束索引 | 新表，无风险 |
| 3 | `accounts` | `AddIndex accounts_dt_user_manager_idx`、`AddIndex accounts_user_dingtalk_idx` | 在线加索引；表小，锁窗口可忽略 |

- 新 app 需登记：`INSTALLED_APPS` 加 `easyauth.notify`（`config/settings/base.py`），
  `CELERY_IMPORTS` 加 `easyauth.tasks.notify`（同文件 316-325 行区段）。
- 三个迁移相互独立，可任意顺序上线；全部先于 API 视图代码发布（见第 7 篇灰度步骤）。
- 回滚：API 未发布前迁移可直接 `migrate <app> <prev>` 逆向；`notify` 表建后一旦有数据，
  回滚只关闭 API 路由与 beat 任务，不删表。

## 6. 保留期

- `NotifyMessage` / `NotifyRecipient`：默认保留 **180 天**
  （`EASYAUTH_NOTIFY_RETENTION_DAYS`），由第 3 篇 §6 的 beat 清理任务按
  `created_at` 批量删除（先删 recipient 后删 message，或直接依赖级联）。
  审计事实不受影响——`AuditLog` 另有自己的保留策略。
- `AppCapability`：配置数据，永久保留。
