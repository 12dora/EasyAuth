# ADR-001：业务授权运营边界

## 状态

已接受

## 日期

2026-06-06

## 背景

EasyAuth 已经明确采用单公司内部部署的集中式授权层定位。Authentik 是登录身份、OIDC subject 和用户在职状态的权威来源；DingTalk 只提供审批流程；EasyAuth 是已连接内部应用的业务授权事实来源。

产品需求要求补齐应用负责人、员工、应用开发者和系统管理员的运营能力，包括权限模板、配置完整性、凭据运营、联调测试、员工权限可见性和失败恢复。该需求容易与 IAM、SSO、DingTalk 流程设计器或下游应用接入平台混淆，因此需要明确架构边界。

## 决策

业务授权运营增强只扩展 EasyAuth 对既有授权模型的配置、展示、联调和排障能力，不改变核心事实来源。

具体决策：

- 公共权限查询 API 保持不变，继续使用 `GET /api/v1/apps/{app_key}/users/{user_id}/permissions`。
- 权限模板和 `PermissionGroup` 只用于 EasyAuth 管理和展示，不出现在公共 API 的 `permissions` 响应中。
- `Permission.key` 仍是下游应用消费的稳定能力标识；模板导入不能改变既有 key 的业务含义。
- 授权事实仍只能通过 RolePermission、AccessRequest、审批结果和 `GrantService` 产生。
- 管理端 API 使用 `/console/api/v1/` 命名空间，只供 EasyAuth 控制台使用，不作为下游应用契约。
- 管理端 API 使用 Django session、CSRF 和 EasyAuth 管理权限，不接受下游 app token 作为管理身份。
- 应用负责人只能管理自己负责 App 的业务授权配置，不能管理 Authentik Provider、Source、Flow、MFA、公司级用户组或 DingTalk 审批模板。
- 系统管理员负责全局审计、失败恢复、紧急撤权和应用负责人归属。

## 备选方案

### 把权限模板作为新的授权事实

优点：

- 模板导入后可以看起来更自动化。

缺点：

- 会让模板、RolePermission 和 AccessGrant 同时影响授权事实。
- 下游应用难以判断权限变化来源。
- 容易绕过审批和 `GrantService`。

结论：拒绝。权限模板只能定义能力目录，不能授予用户权限。

### 在 EasyAuth 中管理 Authentik 应用和 Provider

优点：

- 应用负责人可以在一个界面中配置登录和授权。

缺点：

- 会让 EasyAuth 变成 Authentik 管理前端。
- 登录权限与业务授权边界混淆。
- 需要复制 Authentik Source、Provider、Flow、Policy 等配置语义。

结论：拒绝。Authentik 继续负责身份和登录，EasyAuth 只负责业务授权。

### 让管理端 API 复用 `/api/v1/`

优点：

- 路径更少。

缺点：

- 容易被下游应用误认为稳定接入契约。
- 管理端 session 权限与 app token 权限语义不同。
- 后续兼容性压力更大。

结论：拒绝。管理端 API 使用 `/console/api/v1/`，与下游公共 API 隔离。

### 让 DingTalk 审批通过直接授予权限

优点：

- 流程更短。

缺点：

- DingTalk 会变成事实授权源。
- 审批回调重放、失败重试和版本递增难以统一。
- 无法保证所有授权写入都有 EasyAuth 审计和事务边界。

结论：拒绝。DingTalk 只提供流程通过证据，授权落库仍由 `GrantService` 完成。

## 后果

- 下游应用接入契约稳定，权限模板和运营控制台不会破坏既有权限查询。
- EasyAuth 的授权事实来源保持单一，便于审计、撤权和失败恢复。
- 管理控制台需要实现明确的 AppMembership 和 session 权限校验。
- 权限模板导入需要额外处理历史 Permission 的废弃、禁用和归属迁移。
- 后续如需 SDK、批量授权或周期复核，应继续遵守该边界。

## 阶段落地影响

- `OPS-1` 可以新增管理控制台私有 API、权限模板和联调能力，但不能改变公共权限查询 API。
- `OPS-2` 可以增强员工门户可见性，但不能让员工直接写入授权事实。
- `OPS-3` 可以增加失败恢复和紧急撤权，但必须复用 `GrantService` 并写审计日志。
- `OPS-4` 可以扩展变更、撤销和续期申请，但 DingTalk 仍只是流程通过证据。

具体阶段目标、验收标准、约束和验证方式以需求、架构和 API 设计文档为准；本 ADR 只约束长期边界。
