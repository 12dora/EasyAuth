# EasyAuth MVP 技术规划草案

## 状态

技术规划草案。本文档尚未选择实现技术栈，因为 MVP 规格将该决策标记为 Ask First。

## 规划修正

本计划修正了初始项目拆解中的主要问题：

- 将全部十个开放问题都作为规划输入，而不是只处理前四个。
- 优先采用纵向 MVP 切片，而不是宽泛的横向阶段。
- 在大量 UI 工作之前先定义权限查询 API 契约。
- 将 DingTalk 回调幂等性和授权记录版本控制列为明确任务。
- 在每个会改变状态的切片中纳入审计日志预期。
- 在请求到查询流程可用之前，保持管理控制台最小化。

## 决策待办

这些决策必须在实施开始前完成：

1. 实现技术栈，以及开发、构建、测试、lint、格式化和类型检查的标准命令。
2. 员工门户和管理控制台是一个具备角色感知能力的 Web 应用，还是两个独立应用。
3. 内部应用凭据策略：静态 token、OAuth2 client credentials，或签名请求。
4. 公共 API 中 `user_id` 的含义：EasyAuth user ID、Authentik user ID、email，或员工编号。
5. Authentik 状态更新策略：webhook、定时同步，或两者都用。
6. 首个试点的 DingTalk 审批模板。
7. 已连接应用中可接受的最大权限撤销延迟。
8. 授权记录是否自动过期，还是在变更、撤销或离职前无限期保留。
9. MVP 中管理员是否需要手动紧急撤销能力。
10. 用于验证一天集成的试点应用。

## 依赖图

```text
决策待办
  -> API 和身份契约
    -> 领域模型和持久化
      -> 权限查询契约测试
      -> 试点应用配置
        -> 员工访问请求
          -> DingTalk 审批生命周期
            -> 授权记录应用、版本控制和审计
              -> 由授权记录支持的权限查询
                -> 试点应用集成包

Authentik 状态更新
  -> 离职清理
    -> 授权记录撤销、版本控制和审计
      -> 权限查询返回无活跃访问权限
```

## 纵向 MVP 切片

### 任务 1：解决技术决策

**说明：** 将决策待办转化为已批准的技术计划，不实施代码。

**验收标准：**
- [ ] 已选择实现技术栈和标准命令。
- [ ] 已选择应用凭据方案和公共 `user_id` 语义。
- [ ] 已选择试点的 Authentik 和 DingTalk 集成模式。
- [ ] 已记录撤销 SLA、授权记录过期策略、紧急撤销范围和试点应用。

**验证：**
- [ ] 该计划不与 `docs/specs/easyauth-mvp.md` 中的 MVP 边界矛盾。
- [ ] 审阅者无需追问缺失的架构决策，就能识别第一个可实施切片。

**依赖：** 无

**预估范围：** S

### 任务 2：优先定义权限查询契约

**说明：** 在构建 UI 流程之前，先规定并测试内部应用权限查询 API，因为一天完成试点集成取决于这个契约。

**验收标准：**
- [ ] `GET /api/v1/apps/{app_key}/users/{user_id}/permissions` 响应形态由类型化请求和响应契约表示。
- [ ] 调用应用身份只绑定到一个 `app_key`。
- [ ] 响应始终包含 `roles`、`permissions`、`version` 和 `expires_at`。
- [ ] 禁用应用以及禁用/离职用户的返回行为由规格定义。

**验证：**
- [ ] 契约测试覆盖成功、无活跃访问权限、禁用应用、禁用或离职用户、缺失凭据、无效凭据和跨应用查询拒绝。

**依赖：** 任务 1

**预估范围：** M

### 任务 3：构建试点应用配置

**说明：** 提供配置一个试点应用及其角色、权限、角色-权限映射和审批规则所需的最小管理能力。

**验收标准：**
- [ ] 管理员可以创建一个活跃试点应用。
- [ ] 管理员可以为该应用配置可请求角色和权限映射。
- [ ] 没有审批规则的角色不可请求。
- [ ] 涉及安全敏感内容的配置变更会写入审计日志事件。

**验证：**
- [ ] 集成测试或管理端冒烟测试可以从空状态配置一个试点应用。
- [ ] 测试证明没有审批规则的角色会从员工请求选项中排除。

**依赖：** 任务 1-2

**预估范围：** M

### 任务 4：实施员工授权记录请求切片

**说明：** 允许已认证员工为试点应用请求角色，并创建一个 `AccessRequest`。

**验收标准：**
- [ ] 员工认证使用 Authentik 作为身份来源。
- [ ] 员工可以看到试点应用的可请求角色。
- [ ] 提交请求会创建一个 `AccessRequest`，其中包含正确的请求人、应用、请求角色和初始状态。
- [ ] 请求提交会写入 `access_request_submitted` 审计事件。

**验证：**
- [ ] 端到端冒烟测试覆盖登录、应用选择、角色选择和请求提交。
- [ ] 单元测试覆盖无效角色选择和不可请求角色。

**依赖：** 任务 3

**预估范围：** M

### 任务 5：添加带幂等性的 DingTalk 审批生命周期

**说明：** 将已提交请求连接到 DingTalk 审批，并安全处理审批回调。

**验收标准：**
- [ ] 请求提交会创建一个 DingTalk 审批实例，并存储 process instance ID。
- [ ] 审批回调作为不可信外部输入进行验证。
- [ ] 重复回调不会创建重复授权记录，也不会产生重复终态转换。
- [ ] 被拒绝的审批不会创建或更改授权记录。
- [ ] 审批创建、审批通过和审批拒绝都会写入审计事件。

**验证：**
- [ ] Mock DingTalk 测试覆盖审批通过、审批拒绝、重复回调、格式错误回调，以及未知 process instance 的回调。
- [ ] 测试证明回调处理对重复投递具备事务安全性。

**依赖：** 任务 4

**预估范围：** M

### 任务 6：应用授权记录、版本、缓存 TTL 和审计

**说明：** 将已批准请求转换为用户在试点应用中的当前授权状态。

**验收标准：**
- [ ] 已批准的授权记录请求会创建或更新一个活跃 `AccessGrant`。
- [ ] 授权记录创建、变更或撤销会递增受影响的授权记录版本。
- [ ] 权限查询结果反映最新授权记录状态，并包含可配置的 `expires_at`。
- [ ] 授权记录应用失败会使请求停留在 `grant_failed`，并写入审计事件。

**验证：**
- [ ] 测试覆盖首次授权记录、已变更授权记录、被拒绝请求、授权记录应用失败、版本递增和缓存 TTL 边界。
- [ ] 端到端冒烟测试覆盖请求获批后成功进行权限查询。

**依赖：** 任务 2 和 5

**预估范围：** M

### 任务 7：实施 Authentik 离职清理

**说明：** 当 Authentik 将用户标记为禁用或离职时撤销授权记录，并使后续应用查询返回无活跃访问权限。

**验收标准：**
- [ ] Authentik 仍然是禁用和离职状态的来源。
- [ ] 禁用或离职用户的所有活跃授权记录都会被撤销。
- [ ] 撤销会递增受影响的授权记录版本。
- [ ] 离职清理会写入 `user_departure_detected` 和授权记录撤销审计事件。

**验证：**
- [ ] Mock Authentik 测试覆盖禁用和离职状态变化。
- [ ] 清理后的权限查询返回空 `roles` 和 `permissions`，并带有新的版本和缓存过期时间。

**依赖：** 任务 6

**预估范围：** M

### 任务 8：分别添加访问变更和撤销流程

**说明：** 扩展可用的授权记录流程，使员工可以变更现有访问权限并请求移除访问权限，同时不绕过审批规则。

**验收标准：**
- [ ] 变更请求会创建一个由审批支持的目标角色集合。
- [ ] 撤销请求会创建一个由审批支持的移除请求，除非已批准的技术计划明确允许管理员紧急撤销。
- [ ] 已批准的变更和撤销会更新授权记录、递增版本并写入审计事件。

**验证：**
- [ ] 测试覆盖变更审批通过、变更审批拒绝、撤销审批通过、撤销审批拒绝，以及每种结果后的权限查询结果。

**依赖：** 任务 7

**预估范围：** M

### 任务 9：打包试点集成

**说明：** 提供一个试点应用在一个工作日内完成连接所需的文档和 mock 集成。

**验收标准：**
- [ ] API 文档包含认证、端点形态、响应示例、错误、缓存规则和版本语义。
- [ ] 一个最小 mock 或示例内部应用展示权限查询流程。
- [ ] 试点应用设置步骤从凭据到首次成功查询都有记录。

**验证：**
- [ ] 针对本地或测试 EasyAuth 实例运行示例集成。
- [ ] 手动 QA 证明完整流程：配置应用、请求角色、在 mock 或已配置的 DingTalk 中审批、查询角色和权限。

**依赖：** 任务 1-8

**预估范围：** M

## 检查点

### 检查点：契约就绪

完成任务 1-2 后：

- [ ] 技术决策已获批准。
- [ ] 权限查询 API 契约可测试。
- [ ] 应用认证和应用隔离规则明确。

### 检查点：首个授权闭环

完成任务 3-6 后：

- [ ] 可以配置一个试点应用。
- [ ] 员工可以请求访问权限。
- [ ] 审批会创建或更新授权记录。
- [ ] 试点应用可以查询生成的角色和权限。
- [ ] 每个会改变状态的步骤都有审计事件。

### 检查点：试点就绪

完成任务 7-9 后：

- [ ] 离职清理会撤销活跃授权记录。
- [ ] 变更和撤销流程可用。
- [ ] 试点集成包完整。
- [ ] 端到端手动 QA 覆盖 MVP 成功标准。

## 风险和缓解措施

| 风险 | 影响 | 缓解措施 |
| --- | --- | --- |
| 未解决的 `user_id` 语义 | 高 | 在 API 实施前决策，并记录映射规则。 |
| 应用凭据薄弱 | 高 | 从契约优先开始；测试缺失、无效和跨应用凭据。 |
| DingTalk 重复回调 | 高 | 使用幂等键、唯一 process instance 约束和事务性状态转换。 |
| 审计缺口 | 高 | 将审计事件作为每个状态变更任务的验收标准。 |
| 缓存 TTL 对撤销 SLA 来说过长 | 中 | 将应用 TTL 配置绑定到已批准的撤销 SLA，并测试最大 TTL 边界。 |
| 管理 UI 范围蔓延 | 中 | 将早期管理工作限制为试点应用配置。 |
| Authentik 状态滞后 | 中 | 在实施清理前选择 webhook、定时同步，或两者都用。 |

- 决定授权记录过期是否包含在 MVP 中，还是延后。
